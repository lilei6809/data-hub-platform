# CDP DataSource Management Platform — 技术栈决策文档 v1.0

---

## 一、运行时与框架层

### Java 21

核心选型理由是 **Virtual Threads**。CDP 平台的性能瓶颈是 IO 密集型操作——向数千租户的外部数据源建立连接、执行健康探测、处理查询响应。传统平台线程模型下，每个阻塞 IO 操作占用一个线程（约 1MB 内存），高并发时线程池打满是常见故障模式。Virtual Threads 将阻塞 IO 的成本降至接近零，线程挂起时不占用平台线程，调度器自动恢复。

两个重要认知边界：Virtual Threads 让阻塞变便宜，但不消除下游资源限制。数据库连接数仍受 PostgreSQL `max_connections` 约束，连接池依然必要。响应式编程（WebFlux / Reactor）在 Java 21 之后的新项目中不再有必要引入，维护成本远高于 Virtual Threads。

---

### Spring Boot 3.x

`spring.threads.virtual.enabled=true` 一行配置即可启用 Virtual Threads 支持。提供完整企业级生态：依赖注入、配置管理、Actuator 健康检查、GraalVM Native Image 支持。国际 B2B SaaS Java 服务的实际标准，工程师储备充足。

---

### Spring Data JDBC

CDP 平台的数据访问需要精确控制 SQL。Schema-per-Tenant 要求动态切换 `search_path`，复杂的多租户查询不适合 ORM 的自动映射策略。Spring Data JDBC 保持显式 SQL，没有懒加载、N+1、脏检查等 Hibernate 陷阱，工程师始终清楚每次操作发出了什么 SQL。

不选 JPA/Hibernate 的原因：Schema-per-Tenant 的动态路由与 ORM 自动管理的 Session 上下文存在根本性冲突。不选 MyBatis 的原因：非国际主流，与 Spring Data JDBC 职责重叠，引入两套抽象无必要。

---

## 二、入口网关层

### Envoy Gateway

基于 Envoy Proxy 的 Kubernetes-native 网关，通过 GatewayAPI CRD 声明式管理所有路由与安全策略，与业务代码完全解耦。CNCF 项目，Kubernetes 官方 GatewayAPI 实现之一。完全替代 Spring Cloud Gateway。

**核心 CRD 职责：**

| CRD | 职责 |
|---|---|
| `HTTPRoute` | 路由规则，path / header 匹配，流量分割 |
| `SecurityPolicy` | JWT 验证、API Key 认证、OIDC、CORS |
| `BackendTrafficPolicy` | 超时、重试、Circuit Breaker（并发连接上限）、本地限流 |
| `RateLimitPolicy` | 全局限流，需配合 Redis RLS |

**四条生产级强制约束：**

`defaultAction: Deny`：白名单模式，未被 HTTPRoute 显式允许的路径一律拒绝。`audiences` 必须显式配置，缺失时静默跳过 audience 验证，是高危安全漏洞。`claimToHeaders`：将 JWT Claim（`tenant_id`、`tier`、`feature_flags`）转换为 HTTP Header，下游服务无需解码 JWT，对 Keycloak 完全无感知。客户端伪造 Header 防护：网关层强制删除客户端上送的同名 Header，再注入从 JWT 解析的可信值。

**一个关键认知边界：** Envoy Gateway 的 Circuit Breaker 是并发连接上限（`maxConnections`、`maxPendingRequests`），防止连接数爆炸，不是 Resilience4j 那样的失败率状态机。两者解决不同问题，生产环境同时使用，不能互相替代。

---

### Keycloak

外部身份提供商（IdP），为租户用户、平台管理员、Service Account 颁发 JWT，提供 JWKS 端点供 Envoy Gateway 做 token 签名验证，支持 SAML / OIDC Federation 满足大型企业租户 SSO 对接需求。

Keycloak 不负责 CDP 平台特有的业务属性。`tenant_id`、`tier`、`feature_flags`、`max_datasources` 等字段由 Token Enrichment Service 注入，不存在 Keycloak 中，保持 IdP 的纯净性。

---

## 三、消息中间件层

### Apache Kafka + Spring Cloud Stream

CDP 平台的核心操作（Onboarding、Offboarding、凭证轮换、健康信号处理）是长时间多步骤编排流程，需要持久化事件总线提供 checkpoint 机制和可靠的异步解耦。Spring Cloud Stream 通过函数式编程模型（`Function<I,O>`、`Consumer<T>`）声明式绑定 Topic，屏蔽 Kafka 客户端细节。

**六条生产级强制约束：**

不按 `tenant_id` 动态创建 Topic，所有租户共享业务域 Topic，通过消息体内的 `tenant_id` 字段区分。KEDA 的 `maxReplicaCount` 不能超过 Topic 的 Partition 数，这是硬约束，超出的 Pod 无 Partition 可分配，纯属浪费。EOS（Exactly-Once Semantics）仅保证 Kafka 内部原子性，跨 DB + Kafka 边界必须使用 Outbox Pattern。Batch Consumer 正确签名是 `Function<List<byte[]>, ...>`，`Consumer<List<Message<T>>>` 是无效签名。使用 Confluent Schema Registry 时必须配置 `useNativeDecoding: true`。Consumer Group Lag 是扩缩容的正确信号，CPU 不是。

**核心 Topic 规划：**

| Topic | 生产者 | 主要消费者 |
|---|---|---|
| `tenant.lifecycle.events` | Onboarding / Offboarding Service | Schema Provisioning、Billing、Keycloak Sync |
| `datasource.state.events` | DataSource Config Service | Application Plane Lifecycle BC |
| `credential.rotation.events` | Credential Lifecycle Service | Connection Pool Manager |
| `health.signal.events` | Application Plane Health BC | Health Signal Processor |
| `usage.metering.events` | Application Plane（透明发出） | Usage Metering Service |
| `policy.decision.events` | Policy Engine | Connection Pool Manager |

---

### Strimzi

Kafka on Kubernetes 的 Operator，通过 CRD（`KafkaCluster`、`KafkaTopic`、`KafkaUser`）声明式管理 Kafka 集群，是 EKS 上运行 Kafka 的推荐方式。

---

## 四、数据存储层

### PostgreSQL + Schema-per-Tenant

在单一 PostgreSQL 实例内为每个租户创建独立 Schema（`tenant_{tenant_id}`），通过 `SET LOCAL search_path = tenant_xxx` 实现数据隔离。相比 Row-Level Security 更直观，Schema 间迁移独立互不影响；相比独立数据库实例，共享计算资源成本显著更低。

部署模型对应 Mixed Mode：共享计算（Pool），逻辑隔离数据（Schema-per-Tenant）。

---

### Flyway（per-Schema 模式）

每个租户 Schema 独立执行迁移脚本，Schema 内维护独立的 `flyway_schema_history` 版本表，迁移脚本具备幂等性（`CREATE TABLE IF NOT EXISTS`），支持 Onboarding 中途失败后安全重试。平台版本升级时，Schema Provisioning Service 负责按租户逐批执行迁移，控制批次速率避免写入峰值。

---

## 五、连接池层（双层架构）

这是引入 PgBouncer 后的核心架构决策，两层连接池各司其职。

```
Spring Boot Pod（HikariCP，per-Pod 小池）
    ↓  5~10 个连接 per Pod
PgBouncer（Transaction Pooling，基础设施层）
    ↓  20~50 个真实连接
PostgreSQL
```

---

### PgBouncer

**定位：** 基础设施级连接代理，解决 PostgreSQL 服务端连接数爆炸问题。随着 Kubernetes Pod 数量增加，每个 Pod 各自持有 HikariCP 池，不加控制时真实连接数会超出 PostgreSQL `max_connections` 限制。PgBouncer 在服务端做连接复用，将大量客户端逻辑连接映射到少量真实服务端连接。

**Pooling Mode：** Transaction Pooling。事务结束后连接归还池，是生产环境标准模式，效率最高。Session Pooling 几乎等于没有池，Statement Pooling 不支持多语句事务，均不适用。

**与 Schema-per-Tenant 的兼容性处理：** Transaction Pooling 下，Session 级别的 `SET search_path` 在事务结束后不会重置，下一个租户获取到的连接可能携带上一个租户的 `search_path`，这是跨租户数据泄漏的安全漏洞。

**强制解决方案：** 所有 Schema 切换必须使用 `SET LOCAL`，作用域限定在当前事务内，事务结束自动重置：

```sql
-- ❌ 禁止，Session 级别，Transaction Pooling 下跨租户泄漏
SET search_path = tenant_abc;

-- ✅ 强制，事务级别，事务结束自动重置
BEGIN;
SET LOCAL search_path = tenant_abc;
-- 业务 SQL
COMMIT;
```

所有 Schema 切换逻辑必须在显式事务内，统一封装在 Connection Pool Manager 的连接获取层，不允许散落在业务代码中。

**关键生产配置：**

| 参数 | 值 | 说明 |
|---|---|---|
| `pool_mode` | `transaction` | 事务结束归还连接 |
| `max_client_conn` | 1000+ | PgBouncer 接受的最大客户端连接数 |
| `default_pool_size` | 20~50 | 每个 database/user 的服务端连接数上限 |
| `server_reset_query` | `RESET ALL` | 连接归还时强制重置所有 Session 状态，与 `SET LOCAL` 构成双重保险 |
| `server_check_query` | `SELECT 1` | 连接健康探测 |

**未来分库能力的正确预留方式：** PgBouncer 本身不做路由，不直接支持分库。分库路由能力在 Connection Pool Manager 层实现：维护 `tenant_id → database_url` 映射表，为不同 DB 实例维护独立连接路径，PgBouncer 部署在每个 DB 实例前做连接复用。分库时只需更新路由表，应用代码无感知。

---

### HikariCP

**定位：** JVM 侧连接池，管理 JDBC 连接对象的生命周期。引入 PgBouncer 后，`maximumPoolSize` 相应缩小（5~10 per Pod），避免 HikariCP 自身把 PgBouncer 的连接配额打满。在 Connection Pool Manager 的控制下，支持动态调整 `maximumPoolSize` 和 `minimumIdle`，实现连接池的四档温度管理（HOT / WARM / COOL / SUSPENDED）。

---

## 六、缓存层

### Redis

**四个核心使用场景：**

| 场景 | Key 设计 | TTL 策略 |
|---|---|---|
| Token Enrichment 缓存 | `tenant:{id}:enrichment` | 与 JWT 过期时间对齐，Tier 变更时主动 invalidate |
| 全局 Rate Limit 计数 | 由 Envoy RLS 管理 | 按时间窗口滚动 |
| Connection Pool 状态快照 | `pool:{tenant_id}:{datasource_id}:state` | 无 TTL，由 Pool Manager 主动更新 |
| API Key 验证缓存 | `apikey:{hash}:tenant` | 永久，吊销时主动删除 |

不适合放 Redis 的内容：凭证（绝对不行）、Connector Registry 元数据（变更频率低，PG 足够）、Tenant 核心配置（需要持久化和审计）。

---

## 七、凭证管理层

### HashiCorp Vault

生产级 Secret 管理的行业标准。CDP 平台存储成千上万个租户的数据库密码、API Key、TLS 证书，不能明文存在应用配置或数据库中。提供静态 KV 存储、动态密钥生成、凭证租约管理、完整审计日志。

**存储路径规范：**

```
secret/tenant/{tenant_id}/datasource/{datasource_id}/credentials
```

**两种使用模式：**

静态 KV：凭证写入 Vault，应用通过 Vault API 读取，适合 BYODS 场景（租户提供的数据库密码）。动态密钥：Vault 与 PostgreSQL 集成，按需生成短期临时凭证（如 1 小时有效期的 DB 用户），到期自动失效，安全级别更高，是生产最佳实践。

DataSource Config Service 只存储凭证的 Vault 路径引用（`credential_ref`），不存储任何明文。凭证轮换触发后，Connection Pool Manager 消费事件，用新凭证重建对应连接池。

---

## 八、弹性与容错层

### Resilience4j

应用级弹性库，用于 Application Plane 服务向外部数据源发起操作时的容错控制。

| 模块 | 职责 | CDP 场景 |
|---|---|---|
| `CircuitBreaker` | 失败率状态机（CLOSED / OPEN / HALF_OPEN） | 外部数据源连续失败时自动熔断 |
| `ThreadPoolBulkhead` | 隔离不同租户 / 数据源的执行线程池 | 防止一个慢数据源拖垮全局 |
| `TimeLimiter` | 操作超时强制上限 | 连接探测必须有超时上限，不能无限等待 |

**与 Envoy Circuit Breaker 的本质区别：** Envoy 的 Circuit Breaker 是并发连接数上限，防止连接爆炸。Resilience4j 的 Circuit Breaker 是失败率状态机，基于错误率自动开关熔断器。两者解决不同问题，同时使用，不能互相替代。

---

## 九、自动扩缩容层

### KEDA

Kubernetes HPA 基于 CPU / 内存扩缩容，对 Kafka 消费者服务完全不适用。KEDA 的 `KafkaTrigger` 监听 Consumer Group Lag，Lag 高时自动扩容 Pod，Lag 清零时缩容至最小副本数，是事件驱动服务的正确扩缩容方案。

**硬性约束：** `maxReplicaCount` 绝对不能超过对应 Topic 的 Partition 数。超出 Partition 数的 Pod 无法分配到任何 Partition，处于空闲状态，纯属浪费。生产环境 `minReplicaCount` 建议设为 1，避免冷启动延迟。

## 十、容器编排层

### Kubernetes（EKS）

AWS EKS 托管 Kubernetes，Control Plane 由 AWS 负责，团队只管 Worker Nodes。本地开发使用 Docker Desktop + WSL2 + KIND，完整模拟多节点集群行为，不依赖真实 AWS 环境。

---

### Helm + Flux + Argo Workflows

| 工具 | 职责 |
|---|---|
| Helm | 服务打包，模板化 Kubernetes 资源 |
| Flux | GitOps，持续同步 Git 仓库状态到集群 |
| Argo Workflows | 多步骤有状态流程编排（Onboarding / Offboarding） |

平台层（Platform Onboarding，集群基础设施准备）由 Helm + Flux 管理。租户层（Tenant Onboarding，新租户入驻流程）由 Argo Workflows 编排。两层职责严格分离，不混用。

---

## 十一、可观测性层

### OpenTelemetry（OTel）

以 Java Agent 方式无侵入注入，自动为 Spring Boot、JDBC、Kafka 等组件生成 Trace、Metric、Log 数据。OTel Collector 作为数据中转，将数据路由到不同后端存储。

**关键约束：** Distributed Trace 跨越 Envoy Gateway 边界时，W3C `traceparent` Header 必须由应用代码显式传播，不能假设网关自动完成。

---

### Jaeger（Traces）

分布式链路追踪后端。通过 Trace 可以看到一个 Onboarding 请求经历了哪些服务、每个服务耗时多少、哪一步出现异常，是定位多服务链路问题的核心工具。

---

### Loki（Logs）

结构化日志聚合，与 Grafana 集成。所有服务输出 JSON 结构化日志，每条日志必须携带 `tenant_id`、`datasource_id`、`trace_id` 字段，支持按租户过滤和跨服务关联。

---

### Prometheus + Grafana（Metrics）

指标采集与可视化。核心监控指标：连接池水位（各租户 active / idle 连接数）、Kafka Consumer Group Lag、数据源健康率（ACTIVE vs DEGRADED 比例）、API 响应时间分位数（P50 / P95 / P99）、PgBouncer 服务端连接数水位。

---

## 十三、明确不选用的技术

| 技术 | 不选原因 |
|---|---|
| MyBatis | 非国际主流，与 Spring Data JDBC 职责重叠 |
| JPA / Hibernate | 与 Schema-per-Tenant 动态路由根本性冲突，N+1 风险高 |
| Spring Cloud Gateway | 被 Envoy Gateway 完全替代 |
| WebFlux / Reactor | Java 21 Virtual Threads 场景下无必要引入，维护成本高 |
| Temporal | 功能强大但超出当前阶段需求，Argo Workflows 已足够 |
| RabbitMQ | 吞吐量和水平扩展能力不及 Kafka，不适合平台级事件总线 |
| pgcat | 功能更强但社区成熟度不足（2022 年），PgBouncer 当前阶段已够用 |
| Elasticsearch | 日志用 Loki（成本更低），当前无全文搜索需求 |
